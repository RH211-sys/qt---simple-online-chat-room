```
存在潜在问题，当服务器receive还未完成时就发送query内容，会冲突，这个是题外，可以用锁的思想解决，不用真用锁变量

```

为了调试，先把心跳包检测时间延长



原因应该是缓存读取冲突，建议如果没有读到flag先不读，这里用锁也解决不了

> **socket读取不是线程安全的**。主线程读了 1 字节后，cli_thread 又读 3 字节——这种跨线程读会把 socket 的内部缓冲区状态搞坏。
>
> 服务端 socket 状态坏掉后，`cli->write(response)` 可能静默失败 —— 响应根本没发出去

先测试关闭心跳功能，如果是这个原因，那么要加锁，若不是再找别的

看看write和read有没有返回值(错误码)/错误信号

```
存在，但write返回成功，且用此做了证明，服务端这边没有问题，问题应该还是出在客户端
```

应该定位到问题了：Qt的socket不保证线程安全，跨线程消费和生产可能会损坏socket读/写 状态，这使得尽管代码执行成功，但最终只保留在缓冲区，没有通过socket发出去应该是不同线程同时对一个socket进行写访问(生产/消费 同时进行)，这导致socket 被损坏

那么现在还有两个方案：

- 对所有的消费和生产行为加锁
- 重构，修改出一个单线程消费模块，单线程生产模块

方案选择：

```
 如果改成单线程消费和生产的话，我们需要多增加一个线程+信号和槽，专门用来进行消费，且他要消费所有的协议(文件，消息，kick等等)，这需要再建立大量的信号和槽进行跨线程通信(即别的线程：文件传输，消息模块，主线程等进行peek后立马通知该线程进行read，或通知这个线程进行write)，这要修改的东西特别多
而使用锁的话，只需要在每个消费和生产里面进行保护就行，且相对来讲改动小，不用改架构，
```



所以先用方案1，能不用方案2就不用(这是最坏的打算)
这里让AI帮忙改了，全是体力活

```

// 释放锁，后续broadCast/tellPointedUser/跨客户端write各自加锁
注释改成解耦 消费判断和广播
```

经测试，服务端使用锁来保护socket的方案失效，不过，也有可能是客户端不加锁的原因，或者是服务端的其他原因

现在判断可能是忘记加flush接口的原因

> Qt中，write只是写入到缓冲区，经过flush再发送给客户端

现在这个问题已解决，但最好还是把所有的socket读写都用锁来保护，防止临界区竞态
